**Mistake**: First pass at the FOCAS target function list (Decision-2 brief) included names like `cnc_rdmode`, `cnc_rdtcode`, `cnc_rdsysinfo`, `cnc_rdtoolgrp_id` that look correct but are Series 16/18/21-era names. The FS30i processing DLL (`fwlib30i64.dll`) that serves 0i-MF does not expose them. The extractor flagged them all as NOT FOUND on first run against the real `Fwlib64.h`. → **Rule**: Never add a FOCAS function name to the target list, the spec doc, or the client without first grepping `Fwlib64.h` to confirm it exists. The function-name set differs between FS-16/18/21 and FS30i families, and our docs/training data leak the older names. R9 (FOCAS function name mismatch with reality) is exactly this risk — treat it as a structural hazard, not a one-time mistake.

**Mistake**: Decision-2 brief asked Claude to populate `tasks/spec-focas-calls.md` "verbatim from `C:\Fanuc\FwLib64-runtime\Fwlib64.h`" while the agent was in a Linux container with no Windows mount. → **Rule**: Header / SDK / DLL access lives on the Windows dev box. The agent's job is to write tooling (the extractor) that the operator runs on Windows; the agent does not invent verbatim text it cannot read. Anything claiming to be verbatim from a file the agent didn't read is fabrication.

**Mistake**: First R1 mitigation in `migrations/env.py` checked only `op.schema` — but `CreateForeignKeyOp` stashes `source_schema` and `referent_schema` in `op.kw`, not as direct attributes. A migration creating an FK from `tooling.x` to `tracker.users` walked past the guard silently. → **Rule**: Any guard introspecting Alembic ops must walk both direct attributes ending in `schema` AND keys in `op.kw` ending in `schema`. Spot-check coverage with real op constructors — the text-scraping test harness we had didn't catch this because it didn't exercise FK ops.

**Mistake**: `_check_schema(schema, source)` returned early when `schema is None`, on the assumption "schemaless ops don't need checking." But Postgres routes unqualified DDL through `search_path`, defaulting to `public` — outside our allowlist. A bare `op.create_table('foo')` migration would happily land in `public.foo`. → **Rule**: `schema=None` means "fall through to the connection's search_path default" in Postgres. Treat it as forbidden, not safe. Pair with `SET search_path TO tooling, shared` on the migration connection so unqualified DDL has a safe deterministic landing zone (`tooling`).

**Mistake**: First R1 implementation was autogenerate-only — `process_revision_directives` and `include_object` only fire during `alembic revision --autogenerate`. A hand-written migration with a forbidden FK would never pass through them and would run unchecked at `alembic upgrade` time. → **Rule**: Schema isolation needs layered defense. (1) Runtime DDL inspection via SQLAlchemy `before_cursor_execute` is the primary line — fires on every cursor execute regardless of how the migration was authored. (2) Search-path lockdown on the connection. (3) Autogenerate-time op walker as early-warning. (4) Database-level GRANT as the ultimate belt. Code stops honest mistakes; DB GRANTs stop adversaries.

**Mistake**: First Phase 1 smoke against the real Viper failed with `cnc_allclibhndl3` returning -15 (`EW_NODLL`). The DLLs all existed on disk and `ctypes.WinDLL("Fwlib64.dll")` succeeded — but when the front-end DLL internally `LoadLibrary`'d its sibling `fwlibe64.dll` for the TCP transport, Windows couldn't find it. The first fix attempted `os.add_dll_directory`; the smoke still returned EW_NODLL. → **Rule**: `os.add_dll_directory` adds to the search path consulted by `LoadLibraryEx` with the user-dirs flag. Vendor DLLs that internally use plain `LoadLibrary` (the Fanuc front-end is one) ignore that list. The fix that actually works on Windows: (1) prepend the DLL directory to `PATH` (covers Standard Search Order), (2) preload every sibling DLL by absolute path before loading the front-end (puts them in the process's loaded-module table so subsequent short-name `LoadLibrary` calls find them by name in cache without filesystem lookup), (3) keep `add_dll_directory` as belt-and-suspenders. If a preload itself fails, its OSError names the real missing dependency (typically the MSVC runtime) — a much better diagnostic than the cryptic EW_NODLL coming back from `cnc_allclibhndl3`. The integration smoke caught both passes; it's the only mechanism that exercises the transitive DLL load path. Linux CI can't reproduce because `os.add_dll_directory` is Windows-only and the FOCAS DLLs don't exist on Linux.

**Mistake**: Second Phase 1 smoke against the real Viper got past EW_NODLL but crashed at `read_pots` with EW_NOOPT — and because `run_smoke()` called `source.read_snapshot()` as a single block that propagates exceptions, the entire report was lost. Every successful read before the failure (sysinfo, status, offset_layout, 1600 cnc_rdtofs calls' results) had to be reconstructed from scrollback. → **Rule**: Diagnostic / smoke scripts run against unfamiliar hardware MUST be resilient. Wrap every FOCAS section in its own try/except, accumulate failures into an `errors` list field, and always write the report file at the end regardless of which sections failed. Exit code reflects severity (0 = clean, 1 = some sections failed but report written, 2 = identity / R9 failure). The whole point of a smoke is to capture as much state as possible from one round-trip; a failure in section N must not lose the data from sections 1..N-1.

**Finding** (not a mistake — a real machine fact surfaced by the smoke): the Lance Viper LG-1000AP returns `cnc_rdtofs(num=N, type=1..4)` with rc=4 (`EW_ATTRIB`) for every register and every type. Our `_OFFSET_TYPE_TO_REGISTER_TYPE` mapping (1=h_geom, 2=h_wear, 3=d_geom, 4=d_wear) assumes Memory C offset model. The actual `ofs_type` from `cnc_rdtofsinfo` is the source of truth — if it reports 1 (Memory A), valid type code is 0; if 2 (Memory B), types 1+2 only. → **Rule (open question O2 resolution path)**: read `cnc_rdtofsinfo` first, dispatch the `cnc_rdtofs` calls based on the reported `ofs_type`, document the mapping in spec-focas-calls.md once verified. Never guess offset memory model.

**Finding**: the Lance Viper does NOT have the FANUC tool magazine option licensed (`cnc_rdmagazine` returns rc=6 = `EW_NOOPT`). Pot tracking via `cnc_rdmagazine` is structurally unavailable on this control. → **Rule**: `cnc_rdmagazine` is an OPTION, not a baseline FOCAS function. Architecture must support pot tracking via fallback: either FANUC parameter reads (`cnc_rdparam`) for the pot table parameters, or operator-driven manual pot assignment with no FOCAS sync. Don't hardcode `cnc_rdmagazine` as a required call — feature-detect at startup via a probe call and degrade gracefully if EW_NOOPT.

**Mistake**: Default `assert_expected_control` expected `cnc_type='0i'` and `mt_type='M'`. The Lance Viper's actual `ODBSYS` returns `cnc_type=' 0'` (space + '0', right-justified in the 2-char field) and `mt_type=' M'`. The 'i' in '0i-MF' isn't in `cnc_type` at all — it's encoded in the `series` field as `'D4F1'`. → **Rule**: FANUC `cnc_type` is a 2-char field carrying ONLY the major series number ('0', '30', '16', '21'). The 'i' / 'i-B' / 'i-F' suffixes live in `series`. `_decode_ascii_field` must `strip()` BOTH ends (not just `rstrip`) before comparison. Default identity expectations: `cnc_type='0'`, `mt_type='M'`, `series='D4F1'` for 0i-MF. Documented in `tasks/spec-focas-calls.md` "Verified control identity" section so future contributors don't repeat this.

**Mistake**: Default `DEFAULT_OFFSET_INCREMENT = Decimal("0.001")` was wrong by 10×. Phase 1 panel cross-check on the Viper: panel `H50 = 7.4050 mm`; FOCAS returned `raw=74050`. Correct increment is **0.0001 mm/count**. A 10× scaling error would have written every offset value into the audit log at the wrong magnitude — silent data corruption that would only surface when an operator looked at a UI value and said "that can't be right." → **Rule**: Never assume FANUC offset increment from documentation. Read parameter 1013 at runtime via `cnc_rdparam` (Phase 2 hardening; not yet bound) and refuse to start the poller if the live increment disagrees with the assumed value. Until that's wired, the integration smoke must include an operator-supplied panel value to cross-check at least one register before merge.

**Mistake**: `cnc_rdtofs` type-code semantics for "Memory B" on this 0i-MF differ from FANUC documentation. Standard docs say `type=1 → H_GEOM, type=3 → D_GEOM`. This control has them **swapped**: `type=1 → D_GEOM, type=3 → H_GEOM`. The original code shipped with the docs-implied mapping, which would have labeled every length offset as a diameter offset and vice versa in the audit log. Caught by the panel cross-check (panel "GEOM H" = 7.4050 matched FOCAS type=3, not type=1). → **Rule**: Type-code → register-type mapping is empirically determined per control via panel cross-check at integration time. Hardcoding "the FANUC standard" without verification is the same class of mistake as guessing function names against the wrong control series (R9). Document the verified mapping per-control in `tasks/spec-focas-calls.md`. Final mapping resolved on register 396 (all four panel banks distinct non-zero values): `type=1=D_GEOM, type=2=H_WEAR, type=3=H_GEOM, type=4=D_WEAR but rejected via FOCAS`.

**Finding**: D_WEAR (diameter wear) is structurally unreadable via FOCAS on the Lance Viper. The panel stores and displays it (register 396 panel: `WEAR(D) = 2.0000 mm`) but `cnc_rdtofs(type=4)` returns `EW_ATTRIB` regardless of the value. This is a license/option configuration on the 0i-MF — the FANUC option bit that exposes diameter wear over FOCAS is not enabled. → **Rule**: Document this as a Phase-1 product non-goal for any control whose `ofs_type=2` matches this profile. The audit log will have no D_WEAR rows for this machine; UI must show "N/A — panel only" for D_WEAR cells; Phase 6 write paths cannot push D_WEAR to FANUC and must keep that workflow panel-only. Diameter-wear edits remain operator-driven at the panel, which is how the shop has always managed wear anyway. Workaround if it ever becomes business-critical: purchase the FOCAS option bit from FANUC.

**Finding** (not a mistake — surfaced by the smoke): the 0i-MF's `cnc_modal(datano=-3, type=1)` does NOT return the live current-T modal value. Panel showed `T0045` as the active tool but FOCAS returned `aux_data=0`. Either the modal selectors are wrong for 0i-MF, or this control reports current T via a different mechanism (e.g., the `mstb` flag on `ODBST` from `cnc_statinfo`). → **Rule (Phase 2 follow-up)**: bind `cnc_rdmacro` or read T from `ODBST.mstb`; verify against the panel before accepting. O1 stays unresolved in Phase 1 — `current_t_number` will be `None` in every snapshot until Phase 2 fixes this.

**Mistake**: Spent the entire O1 follow-up assuming the active T was somewhere in the **NC** layer (modal table, FANUC system macro `#4120`, or one of the documented FOCAS magazine functions). On a random-ATC machining center the active tool is **PMC ladder data**, not NC modal data — the magazine controller (separate microprocessor from the NC kernel) owns "what's in the spindle right now". Probes v1..v6 burned ~9 iterations chasing the NC layer: `cnc_modal` (datano sweeps `-20..20`, `100..130` × type `0,1,-1`) returned 0 every time; `cnc_rdtdiseltool` returned `EW_NOOPT`; no `#4xxx` / `#5xxx` system macro carried the panel head; `cnc_rdmagazine` returned `EW_NOOPT`; all seven secondary magazine functions (`cnc_rdcurmgr` / `cnc_rdcurpot` / `cnc_rdpotinfo` / `cnc_rdmagsts` / `cnc_rdspmaint` / `cnc_rdmgrptool`, plus `cnc_rdmagazine` itself) are not exposed by `fwlib30i64.dll` for this control. → **Rule**: For random-access tool changers (Mighty Viper class), assume HEAD / NEXT live in the PMC R-area as bytes from the start. The ATC ladder writes them; the NC kernel doesn't track them as modal. Reach for `pmc_rdpmcrng` immediately, not as a last resort. The "documented FOCAS function for active tool" set is for non-random ATCs and lathes.

**Mistake**: First PMC sweeps (probes v5 word-mode and v6 byte-mode) tried to find specific tool-ID values (`45`, `83`) by scanning every address in R/D/F/K/E. Result: zero hits — even though the values were definitely there at some moment, the machine was running and tools changed mid-sweep. Each sweep took 30-60 seconds; by the time we finished, head had advanced past the value we were looking for. → **Rule**: When the target value is on a moving machine, **never search for the value** — search for **what changes**. Snapshot/diff (`probe_modal_v7.py`) dumps the full PMC state, waits for a single observed transition, dumps again, and computes byte-level differences. The bytes that changed include the head register, pot table cells the indexer rotated, and a small amount of timer/counter noise. Filtering changed-byte list by "new value matches new panel head" pinpoints the storage regardless of byte/word/BCD encoding. The first dump after this approach: 4 changed bytes total, R321 went 25→85 — instantly identifiable as a tool-number register.

**Finding**: The Lance Viper's PMC ladder reads HEAD/NEXT through a scratch pointer at `R321` that flips between the stable storage at `R325` (NEXT) and `R327` (HEAD). A spot-read of R321 in Python returned different values on consecutive calls (31, then 85) because the ladder cycles through reading the two stable cells; we caught it mid-cycle. The actual storage is `R325` and `R327` — not `R321`. → **Rule**: When two consecutive reads of the same PMC byte return different values from a quiet machine, that byte is a **scratch register** the ladder uses internally — not the persistent state. Look at adjacent stable bytes (small even/odd offsets) for the source-of-truth values. Operator cross-check against the panel disambiguates within seconds.

**Finding** (Mighty Viper-specific OEM binding): On the Lance Viper LG-1000AP (Mighty Viper class, 24-pot random-ATC, FANUC 0i-MF Mate, FS30i processing DLL), the OEM PMC ladder stores the panel-displayed loaded-tool state at fixed R-area addresses: **R327 = HEAD** (tool currently in spindle), **R325 = NEXT** (tool to be called next), both as unsigned bytes carrying tool IDs 0..99. Address discovered by probe sequence v1→v9; confirmed against operator panel for both registers (`HEAD=85=R327`, `NEXT=31=R325`). → **Rule**: This binding is OEM-specific to the Mighty Viper PMC ladder, NOT a FANUC standard. Other random-ATC builders (Mazak, Doosan, Okuma, etc.) almost certainly use different R-addresses on their controls. Re-running `probe_modal_v7.py` (snapshot/diff workflow) is the per-machine identification path. Document the binding per machine in `tasks/spec-focas-calls.md` Verified-control-identity section, and never assume cross-machine.
